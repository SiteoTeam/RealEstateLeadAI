# Auth Middleware

**File**: `scraper-agent/src/middleware/supabaseAuth.ts`

Dual-strategy authentication — handles two types of authenticated users.

---

## Decision Tree

```mermaid
flowchart TD
    REQ[Incoming Request] --> HEADER{Authorization\nheader present?}
    HEADER -- No --> 401A[401 Unauthorized]
    HEADER -- Yes --> SUP{Try Supabase\nAuth.getUser}
    SUP -- Valid --> PLATFORM[req.user = Supabase user\nPlatform Admin]
    SUP -- Invalid --> JWT{Try Agent\nJWT verifyToken}
    JWT -- Valid --> AGENT[req.user = {id, slug, role:'agent'}]
    JWT -- Invalid --> 401B[401 Unauthorized]
    PLATFORM --> NEXT[next()]
    AGENT --> NEXT
```

---

## Strategy 1: Supabase Auth Token

Used by the platform dashboard (`App.tsx`). Supabase issues a JWT when the admin logs in via email+password. The middleware calls `supabase.auth.getUser(token)` to validate it.

- Attaches full Supabase user object to `req.user`
- Used for: all `/api/admin/*`, `/api/leads/*`, `/api/stripe/*` endpoints

## Strategy 2: Agent JWT

Used by individual agent admin panels (`/w/:slug/admin`). A custom JWT is generated by `services/auth.ts` when an agent logs in with their password.

- Attaches `{ id, slug, role: 'agent' }` to `req.user`
- Used for: `/api/agent/*` endpoints

---

## `AuthenticatedRequest` Interface

```typescript
interface AuthenticatedRequest extends Request {
    user?: SupabaseUser | AgentJWTPayload;
}
```

Routes check `req.user` to get identity and enforce ownership (agents can only access their own slug).
